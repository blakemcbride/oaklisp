Build instructions for Oaklisp
==============================

Platform Support
================

Oaklisp builds and runs on:

- **Linux** (GCC or Clang) -- primary development platform
- **macOS** (Apple Clang or GCC via Homebrew)
- **Windows / MinGW-w64** (GCC via MSYS2) -- uses POSIX code paths
- **Windows / MSVC** (Visual Studio 2015+) -- uses native Win32 APIs

See `Windows.md` for Windows-specific build details.

Prerequisites
=============

Linux / macOS:
- GCC or Clang
- GNU Make
- Autoconf (>= 2.69) and Automake

Windows (MinGW-w64 via MSYS2):
- mingw-w64-x86_64-gcc, make, autoconf, automake

Windows (MSVC):
- Visual Studio 2015 or later (cl.exe)
- No autotools needed; requires manual config.h (see `Windows.md`)

Optional:
- LaTeX (texlive, latexmk) for building documentation
- gcc-multilib / glibc-devel.i686 for 32-bit mode on a 64-bit host

Quick Start (Linux / macOS / MinGW)
====================================

	autoreconf --install
	./configure
	make
	make install

On a 64-bit host this builds a native 64-bit Oaklisp with 62-bit
fixnums.  No prebuilt binary world is needed; the build uses
oak-cold-linker (built from C) to link portable prebuilt .oa files
from prebuilt/src/world/ into a cold world.

For further details read INSTALL generated by running autoreconf.

Quick Start (Windows / MSVC)
=============================

MSVC builds do not use autotools.  You need a hand-written `config.h`
and a manual compile command or build script.  See `Windows.md` for
the full procedure, including a `config.h` template and sample `cl`
command lines.

Configure Options (autotools builds)
=====================================

Option              Default   Description
------------------  -------   -----------
--enable-64-bit     yes       Native 64-bit mode
--disable-64-bit              Force 32-bit (-m32)
--enable-docs       yes       Build LaTeX documentation
--enable-ndebug     yes       High-speed mode (sets -DFAST)
--enable-threads    no        Native thread support with concurrent GC
--with-world=PATH   search    Path to bootstrap oakworld.bin (for recompiling .oa files)
--with-oaklisp=OAK  no        Path to existing oaklisp for bootstrapping

See ./configure --help for details.

What Gets Built
===============

Two components:

file          | purpose
--------------|----------------------------------------------
oaklisp       | the executable / emulator / virtual machine
oakworld.bin  | pre-built Oaklisp system (read in by the executable)

The world is built in stages from .oak source files:

	new.cold        Cold world (core runtime)
	oakworld-1.bin  Warm boot
	oakworld-2.bin  + macros, bignums, rationals, I/O
	oakworld-3.bin  + compiler (in compiler-locale)
	oakworld.bin    + Scheme compatibility (in scheme-locale)

Thread Support
===============

Thread support is off by default:

	./configure --enable-threads
	make

This enables both `-DTHREADS` and `-DUSE_MARK_SWEEP` (the
mark-sweep GC is required for threaded operation).

On Linux and macOS, threads use pthreads.  On MinGW, threads use
winpthreads (bundled with MinGW-w64).  On MSVC, threads use native
Win32 APIs (CRITICAL_SECTION, CONDITION_VARIABLE, FlsAlloc,
_beginthreadex).  See `Portability.md` for details.

Portability Headers
====================

The emulator includes several header-only portability layers that
abstract platform differences:

Header          Purpose
--------------  -----------------------------------------------
oak-atomic.h    Atomic operations (C11 / GCC builtins / MSVC intrinsics)
oak-thread.h    Thread primitives (pthreads / Win32 native)
oak-getopt.h    Command-line parsing (system getopt / self-contained on MSVC)
signals.h       Signal/longjmp abstractions (sigaction / signal)

These headers automatically select the correct backend based on the
compiler.  No user configuration is needed.

64-bit vs 32-bit
=================

                     64-bit (default)      32-bit
Pointer size         8 bytes               4 bytes
Fixnum range         62-bit signed         30-bit signed
most-positive-fixnum 2305843009213693951   536870911
Heap memory per ref  8 bytes               4 bytes
Instructions per ref 2 (same)              2
Prebuilt .oa files   architecture-independent (same for both)

World images are not portable across word sizes or endianness.

Forcing 32-bit on a 64-bit host:

	./configure --disable-64-bit
	make

This requires 32-bit development libraries (gcc-multilib on
Debian/Ubuntu, glibc-devel.i686 on Fedora/RHEL).

Bootstrapping
=============

Oaklisp is self-hosting: the system is written in Oaklisp, so building
a new world from .oak source requires an existing world.  To break
this circular dependency, prebuilt .oa files (portable compiled
bytecode, architecture-independent ASCII S-expressions) are included
in prebuilt/src/world/.

The bootstrap chain is:

	prebuilt/*.oa → oak-cold-linker → new.cold → boot → oakworld.bin

oak-cold-linker is a standalone C program (built alongside the
emulator) that links .oa files into a cold world.  No existing
oakworld.bin is needed for a fresh build.

For development, if you have a working Oaklisp (either just-built or
installed), .oak files will be recompiled from source automatically.
If compilation fails, the build falls back to prebuilt .oa copies.

To use a specific world for recompiling .oa files from source:

	./configure --with-world=/complete/path/to/oakworld.bin

If you already have Oaklisp installed, you can just use that:

	make OAK=/usr/local/bin/oaklisp

Running Oaklisp
===============

	# Interactive REPL
	oaklisp

	# With explicit world
	oaklisp --world /path/to/oakworld.bin

	# Evaluate and exit
	oaklisp -- --eval '(+ 1 2)' --exit

	# Compile a file
	oaklisp -- --compile myfile --exit

	# Load a file
	oaklisp -- --load myfile --exit

Emulator options go before --, Oaklisp options go after.

Endianness
==========

Oaklisp binary worlds (.bin) are sensitive to CPU endianness and word
size, but the prebuilt .oa files and oak-cold-linker bootstrap is
architecture-independent.  The cold linker generates the correct
format for the target architecture automatically.

Debian Packages
===============

	dpkg-buildpackage -us -uc

Produces two packages:
- oaklisp      -- emulator + world + man page
- oaklisp-doc  -- PDF documentation + examples

Troubleshooting
===============

"no prebuilt world found"
    Configure did not find an oakworld.bin.  This is normal for fresh
    builds -- the build will use prebuilt .oa files with oak-cold-linker.
    To recompile .oa files from source, use --with-world=PATH.

32-bit build fails with missing headers
    Install 32-bit development libraries:
    sudo apt install gcc-multilib      (Debian/Ubuntu)
    sudo dnf install glibc-devel.i686  (Fedora/RHEL)

Out of memory with --size-heap
    The argument is in units of 1024 refs.  On 64-bit, each ref is 8
    bytes, so --size-heap 4096 means ~32 GB.  The default (~1 MB) is
    fine for most use.

MSVC build fails with missing config.h
    MSVC builds require a hand-written config.h.  See Windows.md for a
    template with the correct defines.

MSVC errors about getopt.h / strdup / isatty
    These are handled by portability headers and compat macros.  Make
    sure you are including "oak-getopt.h" (not <getopt.h>) and that
    config.h is on the include path.  See Windows.md.

Binary World Format
===================

The system is started by executing the emulator, which loads a binary
world.  Different varieties of the system can be obtained by booting
different worlds, and by passing various options like the desired locale.
